# BASE — Builder's Automated State Engine

**Version:** 0.1 (Design Spec)
**Created:** 2026-03-15
**Author:** Chris Kahler
**Status:** Design Phase

---

## What BASE Is

BASE is a workspace orchestration tool for Claude Code. It manages the lifecycle of an AI builder's workspace — scaffolding, auditing, grooming, and maintaining the environment that all other tools operate inside of.

**Position in the tool chain:**
```
BASE (workspace) → CARL (session rules) → PAUL (project builds) → AEGIS (code hardening) → Skillsmith (tool creation)
```

BASE is the outermost layer. Everything else operates inside a BASE-managed workspace.

---

## Core Principle

**Scaffold generates manifest. Manifest drives everything.**

```
/base:scaffold → workspace.json → audit → STATE.md → pulse → groom
                      ↑                                        |
                      └────────────────────────────────────────┘
                              (groom updates manifest if structure changes)
```

The manifest (`.base/workspace.json`) is the single configuration point. All commands read from it. The workspace is self-describing.

## Architecture

### File Structure

```
.base/
├── workspace.json    # Manifest — declares workspace areas, cadences, audit strategies
├── STATE.md          # Health snapshot — drift score, area statuses, last groom
├── ROADMAP.md        # Evolution log — what changed, when, why
├── grooming/         # Weekly groom summaries (compressed history)
│   └── 2026-W11.md   # Example: week 11 groom summary
└── audits/           # Deep audit records
    └── 2026-03-15.md  # Example: tonight's full audit
```

### Workspace Manifest (workspace.json)

The manifest declares what the workspace contains and how each area should be maintained. Generated by scaffold, read by all other commands.

```json
{
  "workspace": "my-workspace",
  "created": "2026-03-15",
  "groom_cadence": "weekly",
  "groom_day": "friday",
  "areas": {
    "area-name": {
      "type": "working-memory | directory | config-cross-ref | system-layer | custom",
      "description": "Human-readable purpose",
      "paths": ["file-or-directory-paths"],
      "groom": "weekly | bi-weekly | monthly",
      "audit": {
        "strategy": "staleness | classify | cross-reference | dead-code | pipeline-status",
        "config": {}
      }
    }
  }
}
```

### Audit Strategies (Reusable)

Each area type maps to an audit strategy. Strategies are workspace-agnostic — the manifest tells the audit WHAT to apply them to.

| Strategy | What It Does | Applies To |
|----------|-------------|-----------|
| `staleness` | Check file age, flag if past threshold | Working memory files (ACTIVE.md, BACKLOG.md) |
| `classify` | List items, classify as active/archive/delete | Directories with lifecycle (projects/, clients/) |
| `cross-reference` | Compare directory contents against config file | Tools with config (.mcp.json, package.json) |
| `dead-code` | Scan for unused/orphaned files | System directories (hooks, commands, skills) |
| `pipeline-status` | Check workflow stage progression | Content pipelines, task queues |

Strategies can be extended. Custom strategies for specific workspace types.

## Commands

### /base:scaffold
**Purpose:** Set up BASE in a new or existing workspace.
**Flow:**
1. Scan workspace directory structure
2. Detect common patterns (ACTIVE.md, .mcp.json, .claude/, projects/, etc.)
3. Ask guided questions: "I see X — want me to track this? What cadence?"
4. Generate `workspace.json` manifest
5. Create `.base/` directory structure
6. Generate initial `STATE.md` from current workspace state
7. Install hooks (pulse check on session start)
8. Run first groom to establish baseline

**Output:** A fully configured BASE installation tailored to this workspace.

### /base:pulse
**Purpose:** Daily session activation. Prime the operator with workspace context.
**Flow:**
1. Read `workspace.json` + `STATE.md`
2. Calculate drift score (days overdue across all areas)
3. Check each area against its groom cadence
4. Present health dashboard
5. Recommend next action (groom if overdue, or proceed normally)

**Output:** Conversational workspace briefing. "Here's where things stand. Here's what needs attention."

### /base:groom
**Purpose:** Structured maintenance cycle. The weekly heartbeat.
**Flow:**
1. Read `workspace.json` to determine areas
2. For each area (ordered by priority/staleness):
   - Present current state
   - Guide operator through review (voice-friendly Q&A)
   - Make updates based on responses
3. Update `STATE.md` with new timestamps
4. Update `maintenance.json` with groom record
5. Write groom summary to `grooming/YYYY-WNN.md`

**Output:** All tracked areas reviewed and current. Groom summary logged.

### /base:audit
**Purpose:** Deep workspace optimization. Quarterly or after major shifts.
**Flow:**
1. Read `workspace.json` to determine areas
2. For each area, run its configured audit strategy
3. Present findings per area with recommendations
4. Operator makes decisions (keep/archive/delete/move)
5. Execute approved changes
6. Update `STATE.md`
7. Write audit record to `audits/YYYY-MM-DD.md`

**Critical:** Phases are dynamic. Generated from the manifest, not hardcoded. A 3-area workspace gets 3 phases. A 12-area workspace gets 12.

### /base:status
**Purpose:** Quick one-liner health check.
**Output:** Drift score + overdue items. No conversation.

### /base:history
**Purpose:** Workspace evolution over time.
**Output:** ROADMAP.md — chronological record of major workspace changes.

## Hooks

### base-pulse-check.py (UserPromptSubmit)
**Fires:** Every session start.
**Reads:** `.base/workspace.json`, `.base/STATE.md`
**Logic:**
- Check days since last groom against configured cadence
- If overdue: inject prompt requiring `/base:groom` before proceeding
- If not overdue: inject lightweight status line (drift score, any warnings)
- Check filesystem timestamps on tracked files to detect drift without relying on session-end hooks

**Critical design choice:** No stop hook. Terminal kills bypass stop hooks, making them unreliable for critical state tracking. Instead, the pulse hook detects what changed by comparing filesystem timestamps against STATE.md records. It doesn't matter how the last session ended — pulse always starts from ground truth.

### base-post-groom.py (PostToolUse — Write to STATE.md)
**Fires:** After groom/audit completes and STATE.md is updated.
**Logic:**
- Update timestamps in workspace.json
- Calculate new drift score
- Write groom summary

## Backlog Integration

BASE redefines how the backlog works. Items have time-based properties enforced by grooming:

| Property | Default | Purpose |
|----------|---------|---------|
| Added | Auto-set | When the item entered the backlog |
| Review-by | Priority-based (H:7d, M:14d, L:30d) | When someone must decide on this item |
| Staleness | 2x review-by | Auto-archive threshold — decide or it dies |

During `/base:groom`, backlog items past review-by are surfaced with "decide or kill" pressure. Items past staleness threshold are auto-archived with a note. Nothing sits in limbo indefinitely.

## Distribution Model

BASE is designed to be workspace-agnostic:

- **Solo developer** with one project: 2-3 areas, simple manifest
- **AI builder** (like Chris): 6+ areas, content pipelines, MCP servers, frameworks
- **Agency** managing client work: client directories, shared tools, team workflows
- **Student/learner**: minimal areas, emphasis on scaffold guidance

The manifest adapts. The strategies are reusable. The commands work the same regardless of workspace complexity.

**Installation:** Copy `.base/` scaffold template + hooks into workspace → run `/base:scaffold` → answer questions → operational.

## Relationship to Other Tools

| Tool | Relationship to BASE |
|------|---------------------|
| **CARL** | Both fire hooks on session start. BASE provides workspace-level context, CARL provides session-level rules. BASE could inform CARL which domains to load based on what's active. |
| **PAUL** | PAUL manages individual projects inside the workspace. BASE manages the workspace those projects live in. PAUL phases are linear; BASE lifecycle is cyclical. **Satellite registration:** When `/paul:init` runs in a subdirectory (e.g., `apps/casegate-v2/`), it writes a registration entry to the parent workspace's `.base/workspace.json` satellites section. This makes every PAUL project automatically visible to BASE without manual tracking. Pulse scans registered satellites for staleness/phase status. |
| **AEGIS** | AEGIS audits code quality. BASE audits workspace quality. Different domains, same pattern. |
| **Skillsmith** | BASE itself could be scaffolded via Skillsmith. Skillsmith builds skills; BASE is a skill that manages the workspace. |
| **Ideate** | Ideate seeds new projects. BASE tracks them once they exist. Ideate → projects/ → BASE tracks. |

## Satellite Projects

Projects that live in their own git repos inside the workspace (e.g., `apps/*`) are "satellites." They run their own Claude Code sessions independently. BASE needs visibility into them without owning them.

### Registration Pattern

```json
{
  "satellites": {
    "casegate-v2": {
      "path": "apps/casegate-v2",
      "engine": "paul",
      "state": "apps/casegate-v2/.paul/STATE.md",
      "registered": "2026-01-15"
    },
    "hunter-exotics": {
      "path": "apps/hunter-exotics",
      "engine": "paul",
      "state": "apps/hunter-exotics/.paul/STATE.md",
      "registered": "2026-02-10"
    }
  }
}
```

### How Registration Works

1. **On `/paul:init`**: PAUL checks if a parent workspace has `.base/workspace.json`. If yes, PAUL writes its registration entry automatically.
2. **On `/base:scaffold`**: Scaffold scans `apps/` (or configured satellite directories) for existing `.paul/` directories and auto-registers them.
3. **On `/base:pulse`**: Pulse reads satellite registrations, checks each state file for last activity and current phase. Reports satellite health alongside workspace health.
4. **On `/base:groom`**: Satellites with stale state (no activity in N days) get flagged for review.

### Drift Protection for workspace.json

The manifest itself must not drift. Safeguards:
- **Hooks maintain it** — pulse validates on read, groom updates on write. No manual editing required.
- **Registration is event-driven** — projects register themselves via PAUL init, not via human memory.
- **Scaffold auto-discovers** — running scaffold on an existing workspace re-scans and reconciles the manifest against reality.
- **Groom reconciles** — each groom checks that registered satellites still exist (path valid) and that unregistered projects in satellite directories get flagged for registration.
